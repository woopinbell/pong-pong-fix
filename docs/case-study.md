# Pong Pong 서버 구조 개선

게임 진행과 연결 관리, 결과 저장 책임이 커지면서 서버 내부의 역할을 다시 나눴습니다. 그 과정에서 바꾼 구조와 저장소에서 직접 확인한 결과를 정리했습니다.

## 출발점

게임 진행, WebSocket 연결, 대기열, 타이머와 결과 저장 책임이 한 실행 흐름에 모여 있었습니다. 브라우저가 장기 세션 값을 직접 다룰 여지가 있었고, DB 스키마 생성과 실행 중 마이그레이션도 한 경로로 정리할 필요가 있었습니다. 테스트는 있었지만 실제 PostgreSQL, 별도 프로세스와 브라우저를 나눠 실패 지점을 찾기는 어려웠습니다.

## 개선한 구조

HTTP와 WebSocket 페이로드를 공용 Zod 스키마로 옮겨 API와 웹이 같은 계약을 사용하게 했습니다. 브라우저 인증은 HttpOnly 쿠키로 한정하고, WebSocket에는 30초짜리 일회용 티켓만 전달합니다. 개발 로그인으로는 관리자 권한을 얻을 수 없고, 권한 변경은 시드 또는 운영 명령으로 분리했습니다.

게임 계산은 `PongSimulation`, 시드를 고정한 `PongAi`, `RoomSession`, `Matchmaker`와 스케줄러로 나눴습니다. 서버가 50ms 간격으로 상태를 계산하고 시퀀스가 붙은 스냅샷을 보내므로, 브라우저는 입력만 전송하고 오래된 상태는 버립니다. 재접속 중에는 방과 경기 위치를 15초 보존합니다.

PostgreSQL에는 `Database` 테이블 인터페이스와 행 변환기를 뒀습니다. 경기 결과는 `resultKey` 고유 제약 조건을 기준으로 한 트랜잭션에서 전적, 레이팅 기록, 토너먼트 연결까지 처리합니다. 친구와 토너먼트 동시성 규칙도 DB 제약 조건과 트랜잭션으로 고정했습니다.

비회원 체험은 일반 사용자 데이터와 분리했습니다. 비회원끼리만 매칭하고 6초 뒤 AI로 전환하며, 경기 결과는 DB에 저장하지 않습니다. 생성 횟수, IP별 연결 수, 전체 연결 수와 임시 티켓 수에도 상한을 뒀습니다.

## 확인한 내용

- `shared`와 `web`에 실제 계약·컴포넌트 테스트가 있으며 빈 테스트 통과 옵션을 사용하지 않습니다.
- 메모리 저장소와 PostgreSQL 통합 검사 모두 같은 `resultKey`로 20회 동시에 종료해 결과와 레이팅이 한 번만 반영되는지 확인합니다.
- PostgreSQL 통합 검사에서 마지막 토너먼트 자리에 여러 요청을 동시에 보내고, 두 준결승을 동시에 끝내도 결승이 한 건인지 확인합니다.
- 재현 테스트 데이터는 1,000틱 뒤의 최종 해시를 비교해 같은 시드와 입력의 재현성을 확인합니다.
- 스케줄러 측정은 1, 20, 50, 100개 방을 비교했으며 원본 수치를 `docs/measurements/scheduler-2026-07-24.json`에 보관합니다.

## 확인 범위

정리한 내용은 구현과 로컬 검증 결과에 한정합니다. 운영 처리량, 사용자 지표와 개인 기여 범위처럼 저장소만으로 확인할 수 없는 내용은 포함하지 않습니다.
