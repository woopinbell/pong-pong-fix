# 데이터베이스 불변식

메모리 저장소는 기능 흐름을 빠르게 확인하는 데 유용했지만 동시 실행의 최종 판단자가 될 수 없었다. 같은 경기 종료가 여러 번 들어오거나 토너먼트 마지막 자리에 여러 사용자가 동시에 들어오는 문제는 PostgreSQL 제약과 트랜잭션으로 닫았다.

스키마 변경의 원본은 `packages/db/migrations`의 SQL 파일이다. 애플리케이션 시작 코드나 테스트 문자열에 같은 DDL을 복제하지 않는다. 마이그레이션 적용 여부도 파일 목록과 `kysely_migration`을 비교해 `current`, `pending`, `diverged`로 구분한다.

## 한 번만 끝나야 하는 경기

경기방은 종료될 때 `resultKey`를 만든다. `matches.result_key`의 고유 제약이 같은 결과의 중복 삽입을 막는다. `finalizeMatch`는 충돌이 나면 기존 경기 식별자를 반환하고 사용자 전적과 레이팅을 다시 갱신하지 않는다.

새 결과라면 참가자 행을 식별자 순으로 잠근다. 승자의 승수와 레이팅, 패자의 패수와 레이팅, 양쪽 레이팅 이력을 같은 트랜잭션에 기록한다. 토너먼트 경기라면 해당 대진을 잠그고 결과 연결과 다음 결승 생성도 같은 안에서 처리한다. 중간 단계에서 실패하면 경기 행을 포함한 전체 변경이 롤백된다.

메모리 저장소와 PostgreSQL 통합 검사에서는 같은 `resultKey`로 20번 동시에 종료를 요청했다. 한 결과만 새로 만들어지고 전적과 레이팅도 한 번만 바뀌는 것을 확인했다. 메모리 구현이 같은 결과를 내더라도 PostgreSQL 검사를 대신하지는 않는다.

## 관계의 방향과 정원

친구 관계는 요청자와 수신자의 순서가 바뀌어도 한 쌍이다. `least(requester_id, addressee_id)`와 `greatest(...)` 조합에 고유 인덱스를 두고 두 값이 같은 경우도 제약으로 막았다. 반대 방향의 대기 요청은 새 행을 만들지 않고 수락 상태로 바뀐다.

토너먼트 참가는 토너먼트 행을 잠근 뒤 정원과 다음 시드를 계산한다. `(tournament_id, seed)`가 유일하므로 동시에 계산한 값이 충돌하면 잘못된 두 참가를 모두 인정하지 않는다. 준결승이 동시에 끝날 때 결승 생성도 `(tournament_id, round, slot)` 제약으로 한 건만 남는다.

관리자 차단에서는 사용자 상태와 관리 이력이 하나의 사실이다. 둘을 같은 트랜잭션에 넣어 차단만 되고 이유가 남지 않는 경우를 막았다.

파괴적인 테스트 초기화에도 불변식을 적용했다. `NODE_ENV=test`와 전용 DB 또는 정해진 격리 스키마 형식을 모두 만족한 대상만 초기화한다. 편의 명령도 운영 데이터를 지울 수 있는 예외가 되지 않는다.
