# 빌드 이후의 실행 책임

개발 서버가 뜨는 것과 배포할 수 있는 산출물이 만들어지는 것은 다른 문제였다. 공용 패키지, DB와 API는 실행 JavaScript와 타입 선언을 `dist`에 만들고 SQL 마이그레이션도 함께 복사한다. 웹은 Next.js 독립 실행 서버와 정적 파일을 만든다. 빌드 뒤에는 이 경로들이 실제로 존재하는지 별도 검사한다.

Node.js와 pnpm 버전은 개발 환경, CI와 Docker에서 같게 맞췄다. Dockerfile은 의존성, 빌드와 실행 단계를 분리하고 실행 이미지에는 필요한 산출물만 넣었다. API와 웹은 루트가 아닌 `node` 사용자로 실행한다.

## 시작에는 순서가 있다

PostgreSQL 상태 검사가 통과하면 마이그레이션 전용 컨테이너가 SQL 변경을 적용하고 종료한다. API는 이 작업이 성공한 뒤 시작하며 웹과 Caddy는 API 준비 상태를 기다린다. 서버 시작 함수가 스키마 변경이나 샘플 데이터 생성을 몰래 수행하지 않는다.

`/health/live`와 `/health/ready`도 역할을 나눴다. 프로세스가 살아 있어도 DB 연결이 끊겼거나 마이그레이션이 빠졌거나 종료 중이면 트래픽을 받을 준비가 되지 않은 상태다. 준비 상태 응답에는 내부 예외나 접속 정보를 넣지 않는다.

종료 신호를 받으면 준비 상태를 먼저 내리고 새 매칭을 막는다. 진행 중인 방을 최대 60초 동안 정리한 뒤 WebSocket, HTTP와 DB 풀을 닫는다. 여러 신호가 와도 같은 종료를 두 번 시작하지 않는다.

## 검증 층을 나눈 이유

타입 검사와 단위 검사는 가장 빠른 피드백을 준다. PostgreSQL 통합 검사는 실제 제약과 트랜잭션을 확인한다. 실행 검사는 빌드한 별도 프로세스의 HTTP와 WebSocket을 확인하고, Playwright는 브라우저에서 상태가 연결되는지 본다. 비회원 E2E는 실행 모드와 데이터 경계가 달라 일반 E2E와 분리했다.

Prometheus 지표에는 연결, 대기열, 방, DB 동작, 스냅샷 전달과 누락, 결과 확정, 재접속과 이벤트 루프 지연을 넣었다. 공개 Caddy 경로에서는 `/api/metrics`를 막고 내부 API 경로에서만 수집한다.

운영 가능성은 기능 목록이 아니라 시작, 준비, 관측과 종료가 같은 계약을 지키는지에 달려 있었다. 실행 경계를 코드와 Compose, CI에서 함께 확인한 이유도 여기에 있다.
