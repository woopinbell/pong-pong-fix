# 타입과 런타임 계약

처음 공용 패키지를 만든 이유는 웹과 API에서 같은 타입을 복사하지 않기 위해서였다. 구현이 이어질수록 더 중요한 역할은 네트워크 경계에서 실제 값을 거르는 일이라는 점이 분명해졌다. TypeScript 타입은 빌드 뒤에 사라지고, 브라우저나 소켓에서 들어온 JSON이 그 타입을 지킨다는 보장은 없다.

`packages/shared/src/http.ts`, `game.ts`, `ws.ts`에는 Zod 스키마와 그 스키마에서 추론한 타입을 함께 둔다. 웹은 API 응답을 받은 직후 스키마로 확인하고, API는 경로 매개변수·쿼리·본문과 WebSocket 프레임을 업무 로직에 넘기기 전에 확인한다. 서버가 내보내는 실시간 이벤트도 인코딩 전에 같은 스키마를 통과한다.

## 엄격한 객체가 막아 준 것

값이 필요 없는 요청이라고 해서 검증이 필요 없는 것은 아니었다. 로그아웃 요청의 본문이나 상태 확인 요청의 예상하지 못한 쿼리를 그대로 무시하면 잘못된 호출도 정상 호출처럼 보인다. 빈 자리에는 엄격한 빈 객체를 두고 정의하지 않은 키는 `validation_error`로 끝냈다.

HTTP 오류도 한 모양으로 고정했다. 외부에는 안정된 코드와 메시지, 요청 식별자를 내보내고 필드 오류가 있을 때만 해당 정보를 추가한다. 내부 예외 메시지나 DB 오류가 응답 모양을 결정하지 않는다. 웹의 `ApiError`는 이 구조를 파싱하므로 화면도 상태 코드와 오류 코드를 기준으로 분기한다.

WebSocket 이벤트는 `type`을 판별자로 쓰는 합집합으로 만들었다. 모든 이벤트에는 `v: 1`이 있고, 게임 입력에는 방 식별자와 증가하는 `inputSeq`, `-1 | 0 | 1` 방향만 들어간다. 스냅샷에는 `tick`, 전송 순서인 `sequence`, 서버 시각과 게임 상태가 함께 들어간다. 물리 계산의 시간과 네트워크 전송 순서를 같은 숫자로 취급하지 않았다.

## 계약을 어디까지 공유할지

공용 계약에는 경계를 넘는 값만 넣는다. PostgreSQL 행, Fastify 요청 객체, React 상태와 `GameHub` 내부 방 객체는 공유하지 않는다. 이 객체들까지 공용 패키지에 넣으면 외부 형식이 내부 구현에 끌려간다.

현재 기준은 단순하다. 컴파일 단계의 타입은 개발 중의 사용법을 좁히고, 런타임 스키마는 신뢰할 수 없는 값을 좁힌다. 같은 이름을 쓰더라도 두 역할을 하나로 보지 않는다.
