# 인증과 신뢰 경계

초기에는 기능 연결이 우선이라 브라우저가 세션 값을 직접 다룰 여지가 있었다. 인증 방식을 다시 정리하면서 장기 세션, 짧은 접속권과 일시적인 비회원 신원을 서로 다른 수명으로 나눴다.

일반 사용자 세션은 HttpOnly 쿠키로만 전달한다. 웹의 공통 `fetch`는 `credentials: "include"`를 사용하고 토큰을 로컬 저장소나 JavaScript 상태에 복사하지 않는다. 로그아웃은 브라우저 표시만 바꾸지 않고 저장소의 세션을 삭제한다. 401을 받으면 Query 캐시에서 세션에 묶인 데이터도 함께 정리한다.

## WebSocket에는 새 접속권을 썼다

쿠키나 장기 세션을 소켓 URL에 넣으면 프록시와 접근 로그에 남을 수 있다. 인증된 HTTP 요청으로 32바이트 난수 티켓을 발급하고 수명은 30초로 제한했다. PostgreSQL에는 원문 대신 단방향 해시를 저장한다. 소켓 연결 시 해시를 소비하므로 재사용할 수 없고, 재접속 때도 새 티켓을 받아야 한다.

요청 로그는 쿠키, 인증 헤더, 티켓과 쿼리 전체를 가린다. URL은 경로만 남기고 풀 오류도 제한된 이름과 코드만 기록한다. 관측 가능성을 이유로 다시 비밀값을 노출하지 않게 했다.

역할도 로그인 편의 기능과 분리했다. 개발 로그인은 기존 사용자를 다시 사용하더라도 역할을 일반 사용자로 되돌린다. 관리자 역할은 DB 명령으로 명시적으로 바꾼다. 차단 상태의 사용자는 로그인과 티켓 소비 단계에서 거절된다.

## 비회원은 더 짧고 좁게

비회원은 DB 사용자를 만들지 않는다. 데모 모드에서만 2시간짜리 HMAC 서명 쿠키를 발급하고, 쿠키에는 발급 당시 주소를 함께 넣는다. 요청 주소를 신뢰할지는 `TRUST_PROXY` 설정으로 결정한다. 비회원 티켓과 연결은 프로세스 메모리에만 있고 각각 IP별·전체 상한을 갖는다.

비회원에게 일반 사용자 권한을 하나씩 빼는 방식도 쓰지 않았다. 사용할 수 있는 범위를 경기와 재접속으로 먼저 좁혔다. 결과는 영속화하지 않고 등록 사용자와도 매칭하지 않는다.

신뢰 경계를 정리하면서 남은 원칙은 수명이 긴 인증 수단일수록 노출 지점을 줄이고, 권한이 작은 기능이라도 용량과 재사용 제한을 함께 둔다는 것이다.
