# 기준 부하와 이벤트 루프 지연

실시간 연결과 경기방을 함께 늘렸을 때 서버가 어느 경계에서 느려지는지 확인했다. 아래 값은 프로젝트의 `docs/measurements/load-2026-07-24.json`에 기록한 로컬 실행 결과다.

환경은 macOS `26.5.2 (25F84)`를 실행한 Apple M1 MacBook Air였다. CPU는 8개, 메모리는 17,179,869,184바이트였고 Docker에는 CPU 8개와 8,218,316,800바이트를 배정했다. Docker `28.0.4`, Compose `2.34.0-desktop.1`, k6 `1.6.1`, Node.js `24.18.0`을 사용했다.

## 연결 500개·방 50개

연결 500개 가운데 100개가 50개 방을 만들고 나머지가 연결 상태를 유지하도록 구성했다. 시뮬레이션 간격은 50ms, 스냅샷 간격은 100ms였다. 한 번의 결과에 기대지 않기 위해 새 DB로 스택을 다시 올려 같은 조건을 두 번 실행했다.

두 실행 모두 재접속 성공률만 기준을 넘지 못했다. 최초 연결과 방 생성, 스냅샷 지연, 이벤트 루프 지연, 결과 확정은 기준 안에 있었지만 이 조건을 전체 통과로 기록할 수는 없었다. 2차 실행이 끝난 뒤 서버는 스냅샷 전달 11,231건과 연결 종료에 따른 폐기 5건을 기록했고, API 컨테이너는 재시작 없이 실행 중이었다.

HTTP 요청 시간 p95는 1차 1,477.4189ms, 2차 775.3691ms였다. WebSocket 연결 시간 p95는 각각 548.70888185ms, 2,093.04530835ms로 실행마다 차이가 컸다. 한 번의 지연값보다 두 번 모두 실패한 재접속 단계에 계측을 더 붙이는 편이 먼저였다.

## 혼잡 판정을 고친 전후

수정 전 실행에서는 정상 구간 스냅샷 9,074개 중 253개가 빠졌다. 누락률은 `0.02788186026`, 서버가 교체했다고 기록한 스냅샷은 260개였다. 실제 `bufferedAmount`가 낮아도 `send` 콜백이 늦으면 혼잡으로 판단하던 것이 원인이었다.

혼잡 기준을 소켓의 실제 버퍼 크기로 바꾼 뒤 두 실행 모두 정상 구간 누락이 0건이었다. 콜백 완료 시간은 전달 지연을 재는 값이고 버퍼 크기는 전송 혼잡을 판단하는 값이라는 구분이 필요했다. 수정 전 비교값은 해당 코드의 트리 OID와 함께 결과 파일에 남겼다.

## 다음 확인 범위

이번에는 선택 조건인 연결 1,000개를 다시 실행하지 않았다. 기준 조건의 재접속부터 99%에 미치지 못했으므로 연결 수를 늘려 다른 병목을 찾는 일보다 실패한 티켓 발급, 기존 연결 교체와 15초 방 보존 단계를 나눠 기록하는 편이 순서에 맞았다.

이 결과는 Redis나 다중 인스턴스가 필요하다는 근거도, 필요 없다는 성능 보증도 아니다. 확인된 실패는 재접속 흐름에 있고 분산 세션이나 방 소유권의 한계는 아직 측정되지 않았다. 그래서 인프라를 늘리지 않고 재접속 실패의 재현과 계측을 다음 작업으로 남겼다.
