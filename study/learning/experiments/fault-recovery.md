# DB·공개 경로 장애 복구

정상 부하와 별도로 PostgreSQL과 공개 경로에 장애를 넣어 준비 상태가 실제 의존성 변화를 반영하는지 확인했다. 결과 원본은 프로젝트의 `docs/measurements/fault-recovery-2026-07-24.json`에 보관했다.

Toxiproxy로 DB 응답에 300ms 지연을 넣고 공개 경로에는 150ms 지연을 넣었다. 한 요청의 제한 시간은 5,000ms, 단계별 복구 제한 시간은 15,000ms, 상태 확인 간격은 250ms였다.

| 단계 | HTTP 상태 | 응답 시간 | DB | 마이그레이션 |
| --- | ---: | ---: | --- | --- |
| 기준 상태 | 200 | 9ms | 정상 | 최신 |
| DB 지연 | 200 | 1,540ms | 정상 | 최신 |
| DB 중단 | 503 | 10ms | 중단 | 알 수 없음 |
| DB 복구 | 200 | 21ms | 정상 | 최신 |
| 공개 경로 지연 | 200 | 177ms | 정상 | 최신 |
| 공개 경로 연결 재설정 | 응답 없음 | 18ms | - | - |
| 공개 경로 복구 | 200 | 16ms | 정상 | 최신 |

DB가 느려졌을 때 준비 상태는 성공을 유지했지만 응답 시간에 지연이 반영됐다. DB 연결을 끊자 503과 함께 마이그레이션 상태를 `unknown`으로 내렸고, 프록시를 다시 연 뒤 21ms에 200으로 돌아왔다. 내부 DB 오류나 접속 문자열을 응답에 싣지 않고 상태만 바뀌었다.

공개 경로에 150ms를 넣었을 때 준비 상태 응답은 177ms였다. 연결을 강제로 재설정한 단계는 18ms 뒤 `fetch failed`로 끝났고, 경로를 복구한 뒤 다시 200을 받았다. 네트워크 오류를 HTTP 5xx와 같은 결과로 만들지 않고 별도 관찰값으로 남겼다.

복구가 끝난 뒤 API 컨테이너는 실행 중이었고 종료 코드는 0, 재시작 횟수는 0이었다. 풀 오류 로그에는 `kind`, `errorName`, `errorCode`만 남았으며 접속 문자열과 원본 클라이언트는 노출되지 않았다.

이 실행은 장애가 없을 때의 처리량 검사가 아니다. 의존성이 느려지거나 끊겼을 때 준비 상태가 내려가고, 프로세스를 재시작하지 않아도 다시 올라오는지를 확인한 기록이다.
