# 브라우저의 서버 상태

초기 화면을 만들 때 API가 실패하면 대체 데이터를 보여주는 방식이 편했다. 그러나 빈 데이터, 로딩과 서버 실패가 모두 그럴듯한 성공 화면으로 보였다. 샘플 상태를 걷어낸 뒤 브라우저가 소유하는 값과 서버에서 읽은 값을 나눴다.

HTTP 데이터는 TanStack Query가 맡는다. 내 사용자, 로비, 대시보드, 프로필, 친구, 순위표, 토너먼트와 관리자 목록은 각자의 키를 갖는다. 페이지 컴포넌트는 같은 응답을 별도 `useState`로 복사하지 않는다.

변경 요청 뒤 모든 캐시를 지우는 대신 영향 범위를 적었다. 채팅 전송은 로비, 친구 요청은 친구 목록, 토너먼트 변경은 토너먼트만 무효화한다. 프로필 변경은 이름과 아바타가 나타날 수 있는 사용자, 프로필, 로비, 대시보드, 친구, 순위표와 관리자 키를 함께 갱신한다. 키 목록 자체도 검사해 새 화면을 추가했을 때 빠뜨린 갱신을 찾을 수 있게 했다.

## 인증 만료는 데이터 경계도 바꾼다

공통 API 클라이언트는 응답을 공용 스키마로 파싱하고, 401이면 세션 만료 이벤트를 보낸다. Query 클라이언트는 내 프로필과 친구, 관리자 데이터처럼 세션에 묶인 캐시를 제거하고 `me`를 `null`로 바꾼다. 공개 순위표와 다른 사용자의 공개 프로필은 지우지 않는다.

401은 같은 자격 증명으로 다시 요청해도 해결되지 않으므로 자동 재시도하지 않는다. 일시적인 다른 오류는 한 번만 다시 시도한다. 인증 실패와 네트워크 실패를 한 로딩 상태로 합치지 않았다.

## 게임 연결은 다른 종류의 상태다

게임 스냅샷은 HTTP 캐시보다 수명이 짧고 연결 순서에 의존한다. `GameSocketClient`가 티켓 요청과 소켓 교체를 맡고 reducer가 화면 상태를 만든다. 마지막 순번보다 오래된 스냅샷은 reducer에서 버린다.

Canvas는 스냅샷을 저장소처럼 소유하지 않는다. 최근 두 상태 사이를 보간해 그릴 뿐 판정이나 점수를 계산하지 않는다. 입력도 키보드와 터치가 같은 방향 값으로 모이고, 입력창에 초점이 있거나 화면이 숨겨지면 방향 `0`으로 돌아간다.

브라우저 상태는 서버 응답을 보기 좋게 해석하지만 서버에 없는 성공을 만들어내지 않는다. 이 기준이 로딩, 빈 결과와 실패 화면을 정직하게 유지하는 데 가장 도움이 됐다.
