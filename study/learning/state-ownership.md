# 상태를 소유할 위치

실시간 기능이 늘 때 가장 자주 생긴 문제는 상태가 부족한 것이 아니라 같은 사실을 여러 객체가 조금씩 알고 있다는 점이었다. 초기 `GameHub`는 연결, 대기열, 방, 물리 계산과 결과 저장을 함께 조정했다. 이후 각 상태가 바뀌는 이유를 기준으로 소유자를 나눴다.

| 상태 | 현재 소유자 |
| --- | --- |
| 공·패들·점수·틱 | `PongSimulation` |
| 준비·일시 정지·재접속·종료 | `RoomSession` |
| 대기·매칭·AI 전환 | `Matchmaker` |
| 사용자·방별 입력 순서와 속도 | `InputGate` |
| 활성 방의 시간 진행 | `SharedRoomScheduler` |
| 소켓 교체와 재시도 | `GameSocketClient` |
| 브라우저 경기 화면 상태 | `gameConnectionReducer` |
| 영속 경기 결과 | `MatchResultRepository` |

`GameHub`에는 이 객체들을 연결하는 책임만 남겼다. 소켓 이벤트를 받으면 올바른 소유자에게 명령을 넘기고, 나온 결과를 브로드캐스트하거나 저장한다. 물리 좌표나 방 전이를 직접 계산하지 않는다.

## 계산과 수명 주기를 나눈 효과

`PongSimulation.step`은 입력과 경과 시간만 사용한다. 연결이 끊겼는지, 누가 로그인했는지, 결과를 어느 DB에 저장하는지는 모른다. 반대로 `RoomSession`은 공의 좌표를 모른 채 준비, 일시 정지와 재접속 제한 시간만 관리한다. 물리 계산의 테스트와 연결 수명 테스트가 서로 다른 시계를 사용할 수 있게 됐다.

대기열도 배열을 직접 고치는 대신 `Matchmaker`가 사용자별 `queued`와 `matched` 상태를 갖는다. 경기 종료나 소켓 종료 때 `release` 하나로 상태를 없앤다. 중복 참가를 여러 호출부에서 따로 검사하지 않는다.

브라우저에서도 같은 구분을 유지했다. 소켓 객체는 연결과 재시도를 맡고 reducer는 화면에 보일 상태를 맡는다. HTTP 응답은 TanStack Query가 소유한다. 실시간 스냅샷을 Query 캐시에 넣거나 프로필 응답을 소켓 reducer에 복사하지 않았다.

상태 소유자는 그 상태를 정리할 책임도 갖는다. 타이머를 만든 객체가 타이머를 멈추고, 티켓을 만든 저장소가 소비와 만료를 처리하며, 방을 등록한 스케줄러가 마지막 방에서 실행을 멈춘다. 이 기준이 생긴 뒤 기능 수보다 종료 경로의 수를 먼저 확인하게 됐다.
