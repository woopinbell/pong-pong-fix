# 처음으로 연결된 플레이 흐름

서버가 경기 상태를 계산하고 브라우저가 그 결과를 표시하는 첫 흐름을 만들었다. `GameHub`가 대기열, 방, 준비 명령, 입력과 경기 종료를 한데 조정했고, 게임 계산은 서버의 고정 주기 안에서 실행했다. 클라이언트는 패들 방향만 전송하고 공의 위치나 점수를 보내지 않았다.

웹에는 앱 셸과 내비게이션을 만들고 로비, 경기장, 대시보드, 순위표, 프로필, 토너먼트와 관리자 화면을 연결했다. `PongCanvas`는 서버가 보낸 `GameSnapshot`을 Canvas 2D에 그렸다. DPR을 반영해 논리 좌표와 실제 픽셀 크기를 나누고, 스냅샷 사이를 보간해 서버의 전송 주기가 화면의 그리기 주기와 같을 필요가 없게 했다.

## 빠르게 연결한 만큼 남은 것

첫 화면에는 API 응답이 없을 때 보여주는 대체 데이터가 있었고, 일부 버튼도 화면 상태만 바꿨다. 전체 경로를 확인하는 데에는 도움이 됐지만 실제 서버 상태와 화면에 보이는 상태가 다를 수 있었다. 특히 로그인 전 사용자, 경기 종료 뒤 남은 방 식별자, 정적인 온라인 사용자처럼 성공한 것처럼 보이는 가짜 상태가 남았다.

이 시점의 목적은 완성도가 아니라 HTTP, WebSocket, 브라우저가 한 번에 이어지는지 확인하는 것이었다. Fastify 내부 검사와 별도로 빌드한 API 프로세스를 호출하는 HTTP 실행 검사, 실제 WebSocket 프레임을 주고받는 연결 검사, Playwright 브라우저 흐름을 나눴다. 한 층이 실패했을 때 다른 층의 오류로 가려지지 않게 하려는 구성이었다.

로컬 실행은 PostgreSQL, API, 웹과 Caddy를 Compose로 묶었다. Caddy가 `/api`, `/ws`, 웹 요청을 한 포트에서 나눴고 서비스 상태 검사를 시작 순서에 사용했다. 이 구조는 뒤에서 마이그레이션 작업과 준비 상태를 분리할 때 그대로 확장했다.
