# 운영 안전장치와 마지막 회귀

마지막에는 정상 흐름보다 실패한 뒤 남는 상태를 정리했다. `/health/live`는 프로세스가 요청을 받을 수 있는지만 확인하고, `/health/ready`는 DB 연결과 마이그레이션 상태, 종료 준비 여부까지 확인하게 했다. 종료가 시작되면 준비 상태를 먼저 내리고 새 매칭을 막은 뒤 진행 중인 방을 최대 60초 동안 정리했다. `SIGTERM`과 `SIGINT`가 여러 번 와도 종료 절차는 한 번만 실행했다.

PostgreSQL 연결 풀의 유휴 클라이언트 오류에는 별도 처리기를 달았다. 보고할 값은 `kind`, 안전하게 제한한 오류 이름과 코드뿐이다. 원본 오류 객체, 클라이언트나 접속 문자열을 로그로 넘기지 않았다. 보고 함수가 다시 예외를 던져도 풀의 `error` 이벤트 경계 밖으로 퍼지지 않게 했다.

## 파괴적인 명령과 스키마 변경

테스트 DB 초기화는 `NODE_ENV=test`와 `TEST_DATABASE_URL`이 모두 있을 때만 실행했다. 전용 테스트 DB 이름 또는 정해진 형식의 격리 스키마가 아니면 `Unsafe test reset target`으로 중단했다. 허용된 대상만 스키마를 다시 만들고 전체 마이그레이션을 적용했다.

인증 스키마를 바꾸면서 기존 사용자와 경기 기록은 유지하되 이전 세션은 모두 만료했다. `005_expire_legacy_sessions.sql`은 세션 행만 지운다. 새 인증 규칙을 적용하면서 해석할 수 없는 장기 세션을 억지로 보존하지 않는 선택이었다.

데모와 프로덕션 모드에서는 UTF-8 기준 32바이트 미만의 `SESSION_SECRET`으로 API가 시작되지 않게 했다. Compose의 DB 비밀번호와 세션 비밀값에도 기본값을 두지 않았다. 실행 환경이 빠진 상태를 개발용 기본값으로 조용히 대신하지 않았다.

마지막 회귀에서는 브라우저 사용자 상태를 실행 환경마다 분리하고, 로비 접속 상태처럼 비동기로 반영되는 값은 명시적으로 기다렸다. 연결을 교체하는 시점에는 이전 소켓의 이벤트가 새 상태에 영향을 주지 않는지도 따로 검사했다. 실패를 숨기는 재시도보다 상태 소유자와 정리 순서를 분명히 하는 쪽으로 마무리했다.
