# PostgreSQL 정합성과 트랜잭션

메모리 저장소로 기능을 빠르게 확인한 뒤 실제 PostgreSQL에서만 드러나는 규칙을 닫았다. DB 행 이름과 애플리케이션 객체를 섞지 않도록 `schema.ts`의 테이블 타입과 `rowMappers.ts`의 변환기를 분리했다. 통합 검사는 Testcontainers가 띄운 PostgreSQL에 격리된 스키마를 만들고 마이그레이션부터 정리까지 한 흐름으로 실행했다.

가장 먼저 손본 것은 경기 종료였다. WebSocket 종료, 재접속 만료와 토너먼트 진행이 같은 경기를 거의 동시에 끝낼 수 있었다. 방마다 `resultKey`를 만들고 `matches.result_key`에 고유 제약을 추가했다. `finalizeMatch`는 경기 행을 먼저 삽입하고 이미 같은 키가 있으면 기존 결과를 반환한다. 새 결과일 때만 사용자 행을 잠그고 승패, 레이팅 이력과 토너먼트 연결을 한 트랜잭션에서 갱신했다.

## 애플리케이션 검사만으로 부족했던 규칙

친구 관계는 요청 방향이 달라도 같은 두 사람의 관계다. 두 사용자 ID의 작은 값과 큰 값 조합에 고유 인덱스를 두고 자기 자신에게 보내는 요청도 DB 제약으로 막았다. 서로 반대 방향의 대기 요청이 만나면 하나의 수락 상태로 합쳤다.

토너먼트 참가에서는 토너먼트 행을 잠근 뒤 현재 인원과 다음 시드를 계산했다. `(tournament_id, seed)`에도 고유 제약을 두었다. 마지막 자리에 요청이 동시에 들어와도 정원을 넘지 않고, 두 준결승이 동시에 끝나도 결승 행은 한 건만 생긴다.

토너먼트 방을 만든 뒤 DB 상태 갱신이 실패하는 경우에는 방과 매칭 상태를 되돌렸다. 관리자 차단도 사용자 상태 변경과 `admin_actions` 기록을 같은 트랜잭션에 넣었다. 한쪽만 성공하면 화면과 관리 이력이 서로 다른 사실을 보여주기 때문이다.

이 과정에서 테스트는 SQL 구현의 부속물이 아니라 동시 실행 규칙을 표현하는 수단이 됐다. 메모리 저장소와 PostgreSQL에서 같은 `resultKey`를 20번 동시에 확정해 결과와 레이팅이 한 번만 반영되는지 확인했다.
