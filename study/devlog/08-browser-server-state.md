# 브라우저의 서버 상태 관리

페이지마다 `fetch`, 로딩 상태와 오류 처리를 따로 두자 같은 데이터가 여러 이름으로 캐시됐다. 프로필을 바꾼 뒤 로비에는 새 이름이 보이는데 대시보드에는 이전 이름이 남는 식의 문제가 생길 수 있었다. TanStack Query를 도입하고 서버에서 읽은 데이터의 키와 갱신 범위를 `apps/web/src/lib/query.ts`에 모았다.

내 사용자, 로비, 대시보드, 내 프로필, 공개 프로필, 친구, 순위표, 토너먼트와 관리자 데이터를 서로 다른 키로 나눴다. 변경 요청 뒤에는 영향을 받는 키만 정확히 무효화했다. 로비 채팅은 로비만, 친구 요청은 친구 목록만 다시 읽는다. 프로필 변경처럼 여러 읽기 모델에 퍼지는 작업은 사용자, 프로필, 로비, 대시보드, 순위표와 관리자 화면까지 명시적으로 갱신했다.

401 응답은 일반 네트워크 오류처럼 재시도하지 않았다. 공통 API 클라이언트가 세션 만료 이벤트를 보내고 Query 클라이언트가 사용자별 데이터를 제거한 뒤 `me`를 `null`로 바꿨다. 공개 순위표와 다른 사용자의 공개 프로필처럼 세션과 무관한 데이터는 그대로 남겼다.

## 실시간 상태는 Query 캐시에 넣지 않았다

게임 스냅샷은 빠르게 바뀌고 연결 수명과 함께 사라진다. HTTP로 읽는 서버 상태와 같은 캐시에 넣지 않고 `GameSocketClient`, `useGameConnection`, reducer가 따로 소유하게 했다. 반대로 프로필, 친구와 토너먼트는 WebSocket 메시지로 임시 수정하지 않고 변경 요청이 끝난 뒤 Query가 다시 읽었다.

이 구분 덕분에 화면 컴포넌트는 원격 데이터의 저장 위치를 직접 결정하지 않게 됐다. 캐시는 데이터를 대신 만드는 곳이 아니라 서버 응답을 재사용하고 갱신 시점을 관리하는 곳으로 한정했다.
