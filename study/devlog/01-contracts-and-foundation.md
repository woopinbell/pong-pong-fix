# 공용 계약과 서버 기반 구성

처음에는 웹과 서버를 따로 확장하기 전에 둘 사이의 데이터 모양부터 고정했다. 사용자, 경기 기록, 친구, 채팅, 토너먼트 응답을 `packages/shared/src/http.ts`에 두고 Zod 스키마에서 TypeScript 타입을 만들었다. 게임 보드와 패들, 공, 점수는 `game.ts`, 실시간 명령과 서버 이벤트는 `ws.ts`로 나눴다.

이 선택은 타입을 한곳에 모으는 것만으로 끝나지 않았다. 네트워크에서 받은 값은 컴파일 시점의 타입을 믿을 수 없으므로 HTTP와 WebSocket 입구에서 같은 스키마로 다시 검사했다. 객체 스키마를 엄격하게 만들어 정의하지 않은 필드가 조용히 들어오는 경우도 막았다.

## 저장소를 먼저 분리한 이유

데이터 계층에는 `AppRepository`를 두고 PostgreSQL 구현과 메모리 구현이 같은 동작을 제공하게 했다. API 라우트가 SQL이나 `Map`을 직접 알지 않게 하니 로그인, 로비, 친구, 경기와 토너먼트 흐름을 빠르게 연결할 때 메모리 저장소를 쓸 수 있었다. 반면 고유 제약과 트랜잭션은 메모리 구현만으로 확인할 수 없다는 한계도 처음부터 남아 있었다.

PostgreSQL의 첫 스키마에는 사용자, 세션, 친구 관계, 경기, 채팅, 토너먼트와 관리자 기록을 함께 넣었다. 범위가 넓었지만 이후 기능이 어느 테이블에 속하는지는 분명해졌다. 스키마 변경은 `packages/db/migrations`의 SQL 파일을 원본으로 삼고 Kysely의 마이그레이터가 이름순으로 적용하게 했다.

## 처음 연결한 API

Fastify 서버는 환경 설정에 따라 PostgreSQL 또는 메모리 저장소를 받았다. 로그인, 로비, 대시보드, 프로필, 친구, 순위표, 토너먼트와 관리자 라우트를 저장소 계약 위에 올렸다. 이때는 전체 흐름을 먼저 움직이는 데 집중했기 때문에 인증과 요청 검사는 뒤에서 더 엄격하게 다시 손봤다.

기반을 만들면서 가장 분명해진 것은 타입, 런타임 검증, 영속성 계약이 서로 다른 책임이라는 점이었다. 같은 데이터 모양을 사용하더라도 코드 작성 단계, 네트워크 입구와 데이터베이스가 보장할 수 있는 범위는 달랐다.
