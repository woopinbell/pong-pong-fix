# 실시간 게임 코어 분리

기능이 늘면서 `GameHub`가 물리 계산, 방 상태, 연결과 결과 저장까지 모두 알고 있었다. 시간에 따라 달라지는 오류를 작은 단위로 재현하기 어려웠기 때문에 계산과 수명 주기를 분리했다.

`PongSimulation.step`은 현재 상태, 양쪽 입력과 경과 시간만 받아 다음 상태를 만든다. 네트워크나 DB를 참조하지 않는다. 기본 경기는 초당 20틱으로 진행하고 3점에 도달하거나 45초가 지나면 종료한다. 공의 속도는 진행 시간에 따라 올라가되 크기를 18로 제한했다. 같은 상태와 입력을 주면 같은 결과를 만들 수 있어 재생 fixture와 단위 검사로 경기 계산을 고정할 수 있었다.

`RoomSession`은 준비, 진행, 일시 정지, 재접속과 종료 전이를 맡았다. 연결이 끊기면 이전 상태를 기억하고 `reconnecting`으로 이동한다. 15초 안에 끊긴 연결이 모두 돌아오면 기다리던 상태로 복구하고, 한쪽만 돌아오지 않으면 상대 승리로 끝낸다. 양쪽이 모두 돌아오지 않으면 승자를 만들지 않고 방을 정리한다.

## 시간과 전송을 별도 문제로 보기

입력에는 사용자·방별 순번을 붙이고 초당 30개, 순간 허용량 8개의 token bucket을 적용했다. 늦게 도착한 입력과 지나치게 빠른 입력을 서로 다른 이유로 버렸다. 연결 상태는 15초마다 ping을 보내고 45초 동안 pong이 없을 때 종료했다.

스냅샷 전송이 밀릴 때 모든 과거 상태를 쌓지 않았다. 소켓의 `bufferedAmount`가 256KiB를 넘으면 최신 스냅샷 한 건만 남기고, 1MiB를 넘거나 혼잡이 5초 이상 이어지면 연결을 닫았다. 전송 콜백이 늦다는 이유만으로 혼잡이라고 판단하지 않고 실제 소켓 버퍼를 기준으로 삼았다.

`GameHub`는 이 객체들을 조정하고 명령을 전달하는 역할로 줄었다. 상태 계산과 연결 정책을 같은 콜백에 넣지 않으면서 각 실패를 따로 검사할 수 있게 됐다. 기능 이름보다 상태가 바뀌는 이유를 기준으로 책임을 나눈 것이 가장 큰 변화였다.
