# HTTP·인증·프로토콜 경계 닫기

기능을 한 번 연결한 뒤에는 입력이 어디에서 신뢰되는지 다시 확인했다. 가장 먼저 API 시작 과정에서 마이그레이션과 샘플 데이터 생성을 빼냈다. 서버 프로세스를 여러 개 띄우거나 다시 시작할 때 스키마 변경과 시드가 따라오면 실행 순서가 불분명해졌다. 마이그레이션, 개발 시드와 데모 시드를 각각 명시적인 DB 명령으로 분리했다.

HTTP 요청과 응답은 `packages/shared/src/http.ts`의 스키마로 통일했다. 경로 매개변수, 쿼리와 본문을 한 번에 검사하고 값이 없는 자리에도 엄격한 빈 객체만 허용했다. 본문 없는 로그아웃 요청이나 정의하지 않은 쿼리가 라우트 로직까지 들어오는 일을 막았다. 오류 응답은 코드, 메시지, 요청 식별자와 필요한 경우의 필드 오류만 내보냈다.

## 브라우저에서 장기 비밀값 빼내기

브라우저에 세션 토큰을 저장하던 흐름은 HttpOnly 쿠키로 바꿨다. 개발 로그인은 일반 사용자 역할만 만들고 관리자 권한 변경은 DB 명령으로 분리했다. 차단된 사용자는 새로 로그인할 수 없게 했고 로그아웃은 서버 세션을 실제로 폐기했다.

WebSocket은 쿠키 값을 URL에 옮기지 않았다. 인증된 HTTP 요청으로 30초짜리 일회용 티켓을 발급하고, 저장소에는 원문 대신 단방향 해시만 남겼다. 소켓 연결이 티켓을 소비하면 같은 값으로 다시 접속할 수 없다. 이벤트에는 `v: 1`을 넣어 지원하지 않는 버전을 연결 단계에서 거절했다.

요청 로그에서는 쿠키, 인증 헤더, 티켓과 전체 쿼리를 가렸다. URL은 경로만 기록하고 내부 예외는 안정된 오류 응답으로 바꿨다. 상세한 원인이 필요해도 인증 정보나 DB 접속 문자열을 외부 응답에 섞지 않는 쪽을 선택했다.
