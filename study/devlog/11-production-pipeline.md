# 배포 산출물과 CI 연결

TypeScript 컴파일이 통과해도 컨테이너에서 실행할 파일이 모두 생겼다는 뜻은 아니었다. 공용 패키지와 DB, API의 `dist`, SQL 마이그레이션, Next.js 독립 실행 서버를 실제 배포 산출물로 정했다. `tests/build-artifacts.mjs`가 이 가운데 필요한 12개 경로를 확인하게 해 빌드 성공과 실행 가능성을 구분했다.

API와 웹 Dockerfile은 의존성 설치, 빌드, 실행 단계를 나눴다. 실행 이미지는 소스 바인드 마운트나 시작 시 설치에 기대지 않고 빌드된 파일만 받았다. API와 웹 프로세스는 `node` 사용자로 실행했다. Node.js `24.18.0`과 pnpm `10.32.1`은 로컬 설정, `package.json`, CI와 Dockerfile에서 같은 버전을 사용했다.

## 시작 순서를 컨테이너 계약으로 만들기

Compose에서는 PostgreSQL 상태 검사가 끝난 뒤 `migrate` 작업을 한 번 실행했다. API는 마이그레이션 작업의 성공을 기다리고, 웹은 API 준비 상태를 기다렸다. Caddy는 API와 웹이 모두 준비된 뒤 `/api`, `/ws`, 나머지 웹 요청을 전달했다. 애플리케이션 시작 코드에서 마이그레이션과 시드를 제거한 이유가 이 순서에서 분명해졌다.

CI는 검사 성격에 따라 네 작업으로 나눴다. 빠른 작업은 타입 검사, 단위 검사, 빌드와 산출물 확인을 맡았다. PostgreSQL 통합 검사는 Testcontainers로 따로 실행했다. 프로세스·브라우저 작업은 PostgreSQL 서비스에 마이그레이션과 개발 데이터를 준비하고 빌드한 API와 웹을 직접 띄운 뒤 HTTP 실행 검사, WebSocket 연결 검사와 일반 E2E를 실행했다. 비회원 E2E는 `APP_MODE=demo`가 필요해 별도 작업으로 두었다.

한 작업에 모든 검사를 이어 붙이지 않으니 실패가 소스 타입, DB 동작, 프로세스 시작, 브라우저 흐름 중 어디에서 났는지 찾기 쉬워졌다. 빌드 성공과 실제 실행 가능성을 같은 결과로 취급하지 않게 된 것도 이때부터였다.
