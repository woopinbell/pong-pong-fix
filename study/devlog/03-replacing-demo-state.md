# 샘플 상태를 실제 상태로 바꾸기

처음 만든 화면은 동작하는 모양을 빠르게 확인하기에는 충분했지만 서버의 실제 상태를 끝까지 반영하지 못했다. 친구 요청, 프로필 변경, 토너먼트 참가, 관리자 상태 변경과 경기 채팅을 실제 API 또는 WebSocket 명령에 연결했다. 로비의 접속자 수와 진행 중인 방 수, 채팅도 정적인 값 대신 현재 연결과 저장된 메시지에서 가져왔다.

일시 정지는 화면 애니메이션만 멈추는 기능으로 둘 수 없었다. 서버 방 상태를 `paused`로 바꾸고 양쪽 클라이언트가 같은 스냅샷을 받게 했다. 대기 중인 사용자가 오래 남는 경우에는 레이팅이 가까운 NPC를 고르고, 화면과 경기 기록에서도 NPC라는 사실을 숨기지 않았다.

## 성공처럼 보이던 상태를 걷어내기

서버 응답을 기다리는 동안 샘플 프로필이나 전적을 보여주면 실패와 로딩을 구분하기 어려웠다. 대체 데이터를 제거하고 로딩, 빈 결과, 인증 대기, 오류를 별도 상태로 표시했다. 대시보드의 연승은 승패 수로 추정하지 않고 최근 경기 순서에서 계산했고, 레이팅 이력이 없을 때는 그래프를 만들어내지 않았다.

경기 화면에서는 키 이벤트가 발생할 때마다 입력을 보내는 대신 하나의 방향 상태를 유지했다. 입력창에 초점이 있거나 문서가 숨겨질 때는 같은 명령 경로로 방향 `0`을 보냈다. 경기 종료 뒤에는 방 식별자를 비워 다음 매칭 버튼이 이전 세션에 묶이지 않게 했다.

세션 종료도 브라우저 상태만 지우는 것으로 끝내지 않았다. 로그아웃 때 서버 세션을 삭제하고, API가 401을 반환하면 사용자별 캐시와 진행 중인 조회를 정리했다. 본문이 없는 POST 요청에는 `Content-Type: application/json`을 붙이지 않도록 공통 API 클라이언트도 고쳤다.

이 단계에서 화면은 서버 상태의 복사본이 아니라 서버 상태를 관찰하는 뷰라는 기준이 생겼다. 로딩과 실패를 그럴듯한 샘플 데이터로 가리는 방식도 함께 버렸다.
