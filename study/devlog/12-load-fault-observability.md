# 부하와 장애를 수치로 확인하기

로컬에서 두 명이 경기하는 것만으로는 스케줄러와 WebSocket 전송 경계를 판단하기 어려웠다. API에 연결 수, 대기열, 활성 방, 스냅샷 전달 지연과 누락, 경기 결과 확정, 재접속 결과를 Prometheus 지표로 추가했다. Node의 `monitorEventLoopDelay`로 이벤트 루프 지연 p95도 같은 시점에 기록했다.

기준 부하는 연결 500개와 방 50개로 정했다. 그중 100개 연결이 50개 경기를 만들고 나머지는 접속 상태를 유지했다. 시뮬레이션은 50ms 간격, 스냅샷 전송은 방마다 100ms 간격으로 뒀다. 경기 참가자는 진행 중 연결을 끊고 새 티켓으로 재접속해 방이 보존되는지까지 확인했다.

처음 측정에서는 소켓의 실제 `bufferedAmount`가 낮은데도 전송 콜백 지연을 혼잡으로 보아 정상 구간 스냅샷 9,074개 중 253개가 누락됐다. 누락률은 `0.02788186026`이었다. 혼잡 판단을 실제 버퍼 크기로 바꾸고 같은 10Hz 전송 조건을 두 번 다시 실행했다. 정상 연결에서 관찰한 10,061개와 11,214개 스냅샷의 순번 누락은 모두 0건이었다. 이 전후 값은 `docs/measurements/load-2026-07-24.json`에 남겼다.

## 기준을 통과한 것과 통과하지 못한 것

두 기준 실행은 각각 93.8초와 94.1초 동안 진행됐다. 최초 연결은 두 번 모두 500/500이었지만 재접속은 96/100, 95/100에 그쳐 99% 기준을 통과하지 못했다. 스냅샷 지연은 각각 p95 7ms·p99 20ms, p95 9ms·p99 22ms였고 이벤트 루프 지연 p95는 26.066943ms, 26.181631ms였다. 경기 결과는 실행마다 50건이 저장됐으며 실패와 중복 확정은 없었다.

이번에는 선택 조건인 1,000개 연결을 다시 실행하지 않았다. 기준 조건부터 재접속 성공률이 흔들렸으므로 연결 수를 더 늘리기보다 실패한 티켓 발급, 기존 연결 교체와 방 보존 단계를 먼저 나눠 볼 필요가 있었다.

현재 결과를 단일 인스턴스 통과 근거로 쓰지 않았다. 다만 실패가 재접속 흐름에서 확인됐고 분산 세션이나 방 소유권의 한계는 드러나지 않았으므로 다중 인스턴스와 Redis도 추가하지 않았다. 구조를 먼저 늘리기보다 재접속 실패를 재현하고 단계별 수치를 남기는 일이 다음 과제로 남았다.
