# 비회원 체험을 별도 경계로 만들기

비회원 체험을 일반 사용자 생성의 축약판으로 만들지 않았다. `APP_MODE=demo`에서만 진입 화면을 열고, DB 사용자나 세션 행을 만들지 않는 별도 인증 경계를 두었다. 서버는 2시간짜리 HMAC 서명 쿠키에 비회원 식별자와 접속 주소를 담고, WebSocket 티켓은 메모리에서만 관리했다.

비회원은 같은 종류의 사용자 또는 AI와만 경기한다. 다른 비회원이 6초 안에 들어오지 않으면 AI로 전환한다. 채팅, 친구, 프로필 변경, 토너먼트, 관리자 기능은 사용할 수 없고 경기 결과도 PostgreSQL 전적이나 순위표에 남기지 않았다. 종료 이벤트에는 `persisted: false`, `matchId: null`, `ratingDelta: 0`을 명시했다.

## 메모리 상태에도 상한 두기

DB에 저장하지 않는다고 비용이 사라지는 것은 아니었다. 생성 요청은 IP마다 분당 10회, 티켓 발급은 분당 30회로 제한했다. 한 IP에서 유지할 수 있는 비회원 연결과 대기 티켓은 각각 4개로 뒀고 전체 연결은 200개, 티켓은 400개로 제한했다. 추적할 IP 수도 10,000개에서 멈추게 했다.

프록시가 전달한 주소는 `TRUST_PROXY`를 켠 실행에서만 신뢰했다. 서명 쿠키의 주소와 티켓 발급 주소가 다르면 인증하지 않았고, 새 티켓을 만들 때 같은 비회원의 이전 티켓을 폐기했다. 연결 lease에는 세대별 식별자를 둬 오래된 소켓의 종료 콜백이 새 연결을 해제하지 못하게 했다.

일반 브라우저 E2E와 비회원 E2E는 실행 모드가 달라 별도 작업으로 나눴다. 비회원끼리의 매칭, AI 전환, 재접속과 결과 비영속성을 실제 브라우저 흐름으로 확인했다. 기능 권한만 줄이는 데서 끝내지 않고 인증 수명과 메모리 사용량까지 함께 제한해야 별도 경계로 볼 수 있었다.
